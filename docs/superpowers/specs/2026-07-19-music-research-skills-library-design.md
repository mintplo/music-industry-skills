# 음악 산업 리서치 스킬 라이브러리 설계

- 작성일: 2026-07-19
- 상태: 구조 승인, 문서 검토 대기
- 벤치마크: [`mattpocock/skills`](https://github.com/mattpocock/skills)의 저장소·호출·점진적 공개 구조
- 비고: 벤치마크 저장소의 내용은 복사하지 않고 구조 원칙만 적용

## 1. 제품 정의

첫 제품은 정형 앨범 보고서 생성기가 아니다. 사용자의 음악 산업 질문을 해석하고, 스킬에 기술된 API·검색·크롤링 도구 중 필요한 조합을 선택하고, 여러 출처의 근거를 합쳐 질문에 맞는 답을 만드는 문서 중심 리서치 스킬이다.

예시는 다음과 같다.

- “코르티스의 새 앨범 정보와 콘셉트, 초기 반응을 조사해줘.”
- “이 아티스트가 이전 컴백과 비교해 어떤 프로모션을 바꿨는지 알려줘.”
- “최근 걸그룹들이 어떤 플랫폼에서 선공개 캠페인을 하는지 조사해줘.”
- “A와 B의 일본 시장 성과를 같은 기준으로 비교해줘.”

질문을 미리 정한 앨범·캠페인·시장 서브 스킬 중 하나에 억지로 넣지 않는다. 하나의 리서처가 질문마다 필요한 **소스 스택**을 구성한다. 소스 스택은 해당 질문에 선택된 공급자, 검색 경로, 근거 기준의 묶음이다.

## 2. 벤치마크에서 가져올 구조 원칙

원본 저장소는 스킬을 “small, easy to adapt, and composable”하게 유지하고, 각 스킬을 `SKILL.md`, UI 메타데이터, 필요할 때만 읽는 보조 문서·스크립트로 구성한다. 사용자 호출 스킬과 모델 자동 호출 스킬을 분리하고, 라우터는 기억해야 할 사용자 호출 스킬이 많아질 때만 둔다. 자세한 참고 자료는 스킬 본문에서 문서 포인터로 공개하며, 스크립트는 반복되는 결정적 작업에만 둔다.

이 프로젝트에는 다음처럼 적용한다.

1. 첫 버전은 모델이 자연어 질문에서 자동으로 선택할 수 있는 하나의 넓은 `research-music` 스킬로 시작한다.
2. 공급자 지식은 코드 레지스트리가 아니라 스킬 안의 문서형 Provider Catalog에 둔다.
3. 질문 유형별 절차는 독립 하위 스킬이 아니라 선택적 recipe 문서로 둔다.
4. 실제로 독립 호출어와 독립 완료 조건이 생긴 기능만 나중에 별도 스킬로 승격한다.
5. 사용자 호출 스킬이 많아져 사람이 기억하기 어려워질 때만 `ask-music` 라우터를 추가한다.

## 4. `research-music` 스킬의 역할

`research-music`은 모델 호출 스킬이다. 사용자가 아티스트, 발매작, 컴백, 프로모션, 판매·차트·영상·SNS 반응, 경쟁 아티스트, 음악 시장 동향을 조사하거나 비교해 달라고 할 때 자동으로 사용한다.

`SKILL.md`에는 모든 질문이 공통으로 거치는 단계만 둔다.

1. 질문을 조사 가능한 정보 요구로 분해한다.
2. 엔터티를 식별하고 모호성을 해소한다.
3. Provider Catalog에서 필요한 소스 스택을 선택한다.
4. 공식·1차 출처부터 조사하고 허용된 폴백을 사용한다.
5. 출처, 관측일, 시장, 기간, 측정 정의를 보존한다.
6. 질문에 필요한 형식으로만 합성하고, 찾지 못한 정보는 찾지 못했다고 표시한다.

완료 조건은 고정 보고서 섹션이 아니다. 질문의 각 요구가 근거 있는 답, 명시적인 확인 불가, 또는 다음 조사에 필요한 접근 조건 중 하나로 닫히면 완료한다.

## 5. 문서형 Provider Catalog

`providers/CATALOG.md`는 공급자 이름을 나열하는 목차가 아니라, 정보 요구를 공급자 문서로 연결하는 문서 포인터다.

```text
아티스트·발매 식별 → MusicBrainz, Wikidata, Spotify, Apple Music
한국 차트·출하     → Circle Chart
일본 차트          → Oricon
영상·채널 반응     → YouTube
콘셉트·프로모션    → 공식 사이트, 보도자료, 웹 검색
비정형 페이지      → 정적 HTML, 브라우저, 허용된 크롤러
```

각 공급자 문서는 다음 정보를 한곳에 둔다.

- 제공 능력과 적합한 질문
- 제공하지 않는 정보와 흔한 오해
- 접근 방식: 현재 도구, 공개 API, 브라우저, 선택적 스크립트
- 인증·환경 변수와 자격 증명 없는 폴백
- 주요 입력·출력 필드와 식별자
- 출처 등급과 인용 방법
- 속도 제한, 이용 조건, 상업 사용 주의
- 장애·누락 시 대체 공급자
- 마지막 확인일

새 공급자는 먼저 `providers/<name>.md`와 `CATALOG.md`의 문서 포인터로 등록한다. 실제 HTTP 호출·인증·파싱이 반복되고 문서만으로 안정적으로 수행하기 어려울 때만 같은 스킬의 `scripts/` 또는 향후 MCP 도구를 연결한다.

## 6. Recipe의 역할

Recipe는 서브 스킬이 아니라 특정 질문에서 자주 쓰는 소스 스택과 조사 순서를 설명하는 외부 참고 문서다. `SKILL.md`는 질문이 해당 branch에 들어올 때만 recipe를 읽도록 명확한 문서 포인터를 둔다.

- `artist-and-album.md`: 아티스트 식별, 카탈로그, 특정 발매작, 콘셉트, 성과
- `release-campaign.md`: 티저·선공개·MV·미디어·오프라인 프로모션 타임라인
- `artist-comparison.md`: 비교 대상, 시장, 상대 기간, 지표 호환성
- `market-trend.md`: 표본 정의, 검색 기간, 반복 패턴과 반례

Recipe가 질문의 범위를 제한하지 않는다. 한 질문이 여러 branch에 걸리면 필요한 문서만 함께 읽어 하나의 소스 스택을 만든다.

## 7. 기존 디스코그래피 작업의 처리

현재 `research-artist-discography`는 최종 제품의 하위 호출 스킬로 강제하지 않는다.

- MusicBrainz, Apple, Circle, YouTube 등의 조사 지식은 공급자 문서의 재료로 이동한다.
- 발매 그룹·판본·역할 분류 지식은 `entity-resolution.md`와 `artist-and-album.md`로 이동한다.
- 기존 수집·파싱 Python 코드는 버리지 않고, 반복되는 결정적 수집이 필요할 때 선택적으로 호출하는 스크립트로 보존한다.
- 고정된 전체 디스코그래피 보고서 형식과 결정론적 보고서 렌더러는 기본 흐름에서 제거한다.
- 새 `research-music` 전방 테스트가 통과한 뒤 기존 스킬은 `deprecated/`로 이동하고 설치 링크를 새 스킬로 교체한다.

## 8. 근거와 출력 원칙

출력 형태는 질문에 따라 달라진다. 짧은 사실 질문에는 짧게 답하고, 복합 비교에는 표·타임라인·차트를 사용하며, 넓은 시장 조사에는 조사 범위와 표본을 먼저 밝힌다.

모든 branch에 공통인 최소 원칙만 유지한다.

- 핵심 사실에는 직접 출처를 붙인다.
- 성과 수치에는 시장, 기간, 측정 정의, 관측일을 보존한다.
- 서로 다른 단위·시장·기간은 합산하지 않는다.
- 공식값과 비공식 공개 신호를 구분한다.
- 접근 제한, 로그인, robots, 이용 조건을 우회하지 않는다.
- 공급자가 답하지 못하는 질문은 다른 공급자로 폴백하거나 확인 불가로 닫는다.

엄격한 구조 검증은 모든 답변의 선행 조건이 아니다. 수치를 직접 비교하거나 CSV·JSON을 내보내는 branch에서만 선택적으로 사용한다.

## 9. 첫 버전 범위

첫 버전은 다음을 포함한다.

- `research-music` 모델 호출 스킬과 Codex 메타데이터
- Provider Catalog와 초기 공급자 문서
- 네 개의 선택적 recipe
- 공통 근거·엔터티·지표 참고 문서
- 기존 디스코그래피 수집 스크립트의 선택적 연결
- 자연어 질문에서 필요한 공급자를 선택하는 전방 테스트

첫 버전에서는 다음을 만들지 않는다.

- 런타임 Provider Registry 코드
- 항상 실행되는 서버·대시보드·스케줄러
- 고정된 앨범 보고서 템플릿
- 모든 질문에 강제되는 JSON 스키마나 보고서 검증기
- 기업용 API·MCP의 실제 인증 연동
- 많은 독립 서브 스킬과 조기 라우터

## 10. 확장 규칙

새 정보원이나 조사 유형은 다음 순서로 확장한다.

1. 기존 Provider Catalog와 recipe로 처리 가능한지 확인한다.
2. 새 공급자라면 공급자 문서와 카탈로그 포인터를 추가한다.
3. 반복되는 결정적 처리가 확인되면 작은 스크립트를 공급자 문서와 함께 둔다.
4. 독립적인 호출어와 완료 조건이 반복해서 확인될 때만 별도 모델 호출 스킬로 승격한다.
5. 사용자 호출 스킬이 여러 개가 되어 인지 부담이 생길 때만 `ask-music` 라우터를 추가한다.

각 의미는 한곳에만 둔다. 공급자 사실은 공급자 문서, 공통 근거 규칙은 reference, 질문별 순서는 recipe, 모든 branch가 거치는 단계만 `SKILL.md`에 둔다.

## 11. 검증 기준

문서형 스킬은 전방 테스트로 검증한다.

- 단일 사실 질문이 불필요한 전체 보고서로 팽창하지 않음
- 코르티스 같은 아티스트·앨범 질문이 카탈로그, 공식 소개, 차트·영상 공급자를 조합
- 캠페인 질문이 앨범 목록에 갇히지 않고 공식 콘텐츠와 기사·웹 자료를 조사
- 비교 질문이 동일 기준을 먼저 정하고 호환되지 않는 지표를 분리
- 시장 질문이 표본과 기간을 밝히고 여러 공급자를 사용
- 자격 증명이 없는 공급자는 폴백 또는 확인 불가로 처리
- 질문에 없는 고정 섹션이나 전체 디스코그래피를 강제로 출력하지 않음

완료 조건은 서로 다른 질문 branch를 최소 한 번씩 새 컨텍스트에서 실행하고, 선택한 소스 스택과 최종 답이 각 질문의 요구에 맞는다는 독립 검토를 통과하는 것이다.
