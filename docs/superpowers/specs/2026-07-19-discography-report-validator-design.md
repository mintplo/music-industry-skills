# 디스코그래피 보고서 결정론적 검증기 설계

- 작성일: 2026-07-19
- 상태: 설계 승인, 문서 검토 대기
- 대상: `research-artist-discography` 스킬 내부 검증·렌더링 단계
- 작업 위치: 기존 `main` 브랜치, 별도 워크트리 없음

## 1. 문제와 결정

기존 스킬은 발매작 분류와 수치 근거 규칙을 문서로 설명하고 마지막에 AI가 스스로 감사한다. 다섯 차례의 독립 전방 테스트에서 다음 오류가 서로 다른 형태로 반복됐다.

- 에디션을 별도 발매 그룹으로 중복 기재
- 여러 리믹스 발매 그룹을 범주 요약 한 행으로 합침
- 최근 N개 후보 원장을 전체 인벤토리보다 일찍 끊음
- 서로 다른 관측 기간의 피크 순위를 같은 기간 비교로 시각화
- 성과 수치와 근거 ID 또는 콘셉트·프로모션 사실과 인접 출처의 연결 누락

규칙 문구를 더 늘리는 방식은 중단한다. 기존 자연어 스킬을 유지하면서 구조적 정확성은 스킬에 포함된 결정론적 검증·렌더링 모듈이 담당한다. 웹 애플리케이션, 서버, 스케줄러 또는 외부 데이터베이스는 만들지 않는다.

## 2. 고려한 접근

1. **프롬프트 지침만 계속 보강:** 변경량은 작지만 이미 반복 실패가 확인되어 채택하지 않는다.
2. **구조화 데이터 검증 후 핵심 표·차트 렌더링:** 작은 인터페이스 뒤에 목록 대조와 지표 비교 규칙을 숨길 수 있어 채택한다.
3. **완전한 수집·저장 파이프라인:** 가장 강력하지만 첫 스킬의 로컬·주문형 범위를 넘으므로 후속 플러그인 단계로 미룬다.

## 3. 모듈과 인터페이스

새 모듈은 순수한 인프로세스 모듈이다. 네트워크나 파일 시스템을 직접 호출하지 않고 Python 객체를 받아 검증 결과와 렌더링 가능한 구조를 반환한다.

```python
validate_report_payload(payload: dict) -> list[ValidationIssue]
render_report_sections(payload: dict) -> str
```

`validate_report_payload`가 오류를 하나라도 반환하면 `render_report_sections`는 결과를 만들지 않는다. 호출자는 내부의 정렬, 집합 대조, 비교 키 계산을 알 필요가 없다. 테스트는 이 두 인터페이스만 사용한다.

CLI 어댑터는 얇게 유지한다.

```text
python scripts/validate_discography_report.py INPUT.json --output SECTIONS.md
```

- 성공: 종료 코드 0, 검증된 Markdown 구조 블록 생성
- 실패: 종료 코드 1, 모든 오류를 stderr에 표시, 출력 파일 미생성
- 기존 `collect_discography_data.py`와 `validate_dataset()`의 스키마·동작은 유지

## 4. 보고서 페이로드

기존 `artist`, `releases`, `observations`를 확장하되 기존 수집 데이터와 충돌하지 않는 `report_schema_version: "1.0"`을 사용한다.

### 4.1 발매 그룹

각 `releases[]` 행은 다음 필드를 가진다.

- `release_group_id`, `title`, `first_release_date`
- `release_type`, `role_status`, `default_included`
- `source_urls[]`, `source_release_group_ids[]`
- `editions[]`: `edition_id`, 제목, 국가·포맷·발매일, 원본 식별자

`first_release_date`는 ISO 날짜 또는 명시적인 `unknown`만 허용한다. 날짜 범위와 `리믹스들` 같은 범주 요약은 하나의 발매 그룹 행으로 사용할 수 없다. 같은 공급자 네임스페이스의 서로 다른 발매 그룹 식별자를 한 행에 넣으면 범주 병합으로 판정한다. 에디션은 부모 행 안에만 존재하며 별도 마스터 행과 중복될 수 없다.

### 4.2 최근 N개 선정

`selection`은 다음을 가진다.

- `requested_n`
- 명시적 `filters`: 기간, 유형, 앨범명
- `candidate_release_group_ids`: 필터 적용 후 고려한 모든 발매 그룹
- `selected_release_group_ids`

검증기는 마스터 인벤토리에서 필터를 직접 적용하고 예상 후보 집합을 계산한다. 후보 원장 집합이 예상 집합과 정확히 같아야 한다. 후보는 최초 발매일 내림차순, `unknown`은 마지막으로 정렬하며, 제외·불확실 역할을 표시한 뒤 첫 N개 기본 적격 행만 선택한다.

### 4.3 수치 근거와 비교

각 `observations[]`에는 고유한 `evidence_id`와 기존 근거 필드 외에 다음 비교 필드를 둔다.

- `period_basis`: 예: `first_complete_chart_week`, `first_7_days`
- `window_length_days`

`comparisons[]`는 `metric_family`, 포함할 `evidence_ids`, 시각화 종류 또는 생략 사유를 가진다. 같은 비교 그룹의 모든 근거는 시장, 지표, 단위, 측정 방식, `period_basis`, 기간 길이가 같아야 한다. 서로 다른 길이의 사용 가능 기간에서 계산한 피크는 직접 비교할 수 없다.

비교 가능한 근거가 둘 이상이면 타임라인과 별개로 막대·선·역순위 텍스트 차트 중 하나를 생성한다. 비교할 수 없으면 구체적인 인접 생략 사유를 렌더링한다. 서로 다른 단위를 합산하거나 종합 점수를 만들지 않는다.

## 5. 렌더링 범위

검증기가 생성하는 Markdown 블록은 다음 네 부분이다.

1. 한 발매 그룹당 한 행인 전체 마스터 인벤토리
2. 전체 기본 발매 흐름 타임라인
3. 최근 N개 전체 후보 원장과 선정 표시
4. 증거 ID가 포함된 지표표와 비교 가능한 차트 또는 생략 사유

AI는 이 블록을 수정하지 않고 최종 브리프에 삽입한다. 콘셉트·프로모션 표와 해석 문장은 AI가 작성하되 기존 인접 직접 인용과 숫자 감사 규칙을 계속 적용한다.

## 6. 오류 처리

검증 오류는 한 번에 모두 반환하며 경로와 수정 가능한 메시지를 포함한다.

- `releases[7].first_release_date: expected ISO date or unknown`
- `selection.candidate_release_group_ids: missing rg-123`
- `selection.selected_release_group_ids: expected first 3 eligible ids`
- `comparisons[0]: period_basis/window length mismatch`
- `observations[4].evidence_id: duplicate value E03`

스킬은 오류가 있으면 조사 데이터를 수정하고 검증을 다시 실행한다. 오류를 숨기거나 검증되지 않은 표를 최종 답변에 사용하지 않는다.

## 7. 테스트와 전방 검증

TDD로 다음 인터페이스 행동을 고정한다.

- 유효한 페이로드가 오류 없이 네 Markdown 블록을 생성
- 중복 발매 그룹·에디션 중복·날짜 범위 거부
- 제외 발매를 생략하거나 후보 원장을 조기 절단하면 실패
- 날짜순 첫 N개가 아닌 선정은 실패
- 존재하지 않거나 중복된 근거 ID는 실패
- 시장·단위·측정 방식·기간 기준·기간 길이가 다른 직접 비교는 실패
- 비교 가능한 데이터의 차트 누락과 비교 불가능 데이터의 생략 사유 누락은 실패
- CLI 실패 시 출력 파일을 만들지 않음
- 기존 수집기·데이터 검증 테스트가 그대로 통과

구현 후 새 컨텍스트에서 실제 아티스트 요청을 한 번 더 실행한다. 전방 결과는 별도 검토자가 스킬 계약과 검증기 사용 여부만 읽고 판정한다.

## 8. 완료 조건

다음 조건을 모두 만족해야 완료한다.

1. 스킬이 최종 보고서 전에 검증 CLI를 필수로 실행한다.
2. 검증된 구조 블록만 최종 답변에 사용한다.
3. 과거 전방 테스트에서 드러난 목록 중복·범주 합치기·후보 누락·기간 불일치가 결정적 테스트에서 재현되고 차단된다.
4. 전체 단위 테스트, 스킬 검증기, 실제 카탈로그 스모크 테스트가 통과한다.
5. 새 독립 전방 테스트와 전체 코드 리뷰에서 Critical 또는 Important 문제가 없다.
