# 음악 산업 리서치 스킬 시리즈 설계

- 작성일: 2026-07-19
- 상태: 대체됨 — `2026-07-19-music-research-skills-library-design.md`가 제품 구조를 대체함
- 첫 산출물: `research-artist-discography` Codex 스킬
- 장기 산출물: `music-industry-research` Codex 플러그인과 전문 스킬 시리즈

## 1. 배경과 목표

엔터테인먼트 마케팅 실무자가 특정 아티스트의 발매 흐름을 조사할 때 전체 앨범 목록부터 각 발매작의 성격, 판매·차트 성과, 공개 SNS 지표까지 여러 출처에서 반복적으로 취합해야 한다. 첫 버전은 별도 웹 애플리케이션이 아니라 Codex에서 자연어로 호출하는 재사용 가능한 스킬로 만든다.

사용자는 다음과 같이 요청한다.

> BTS가 지금까지 발매한 앨범들을 조사하고 시기별 성과와 콘셉트 변화를 보여줘.

> 에스파의 앨범들을 조사하고 최근 세 앨범의 발매 성과와 프로모션 방식을 비교해줘.

스킬은 아티스트 이름만으로 공식 디스코그래피를 식별하고 공개 데이터를 조사한다. 서로 다른 지표의 정의를 보존한 채 디스코그래피 브리프·발매작별 비교표·시각화를 대화 안에서 반환하며, 필요하면 특정 앨범을 상세 탐색한다.

## 2. 확정된 제품 원칙

1. K-pop을 우선 지원하고 글로벌 팝으로 확장한다.
2. 무료·공개 데이터를 우선 사용한다. 회사의 기업용 데이터 계약은 첫 버전의 필수 조건이 아니다.
3. 특정 아티스트의 디스코그래피를 일괄 조사하는 흐름을 먼저 만들고, 정기 수집은 이후 플러그인 단계에서 추가한다.
4. 아티스트의 전체 발매작을 다루는 디스코그래피 브리프를 기본값으로 제공하며 기간·발매 유형 필터와 특정 앨범 상세 탐색도 같은 스킬이 지원한다.
5. 정확한 판매량을 확보하지 못하면 공식 차트 순위와 스트리밍·SNS 대체 지표를 사용한다.
6. 위키·기사의 수치는 공식 수치와 섞지 않고 참고값으로 분리한다.
7. 모든 수치와 해석에는 출처, 기준일, 측정 정의, 신뢰도를 붙인다.
8. 사이트 차단이나 권한 제한을 우회하지 않는다. 데이터가 없으면 없다고 보고한다.

## 3. 제품 구조

### 3.1 첫 단계

`research-artist-discography`를 독립 설치 가능한 Codex 스킬로 만든다. 이 스킬 안에서 다음 세 모드를 지원한다.

- 아티스트 전체 디스코그래피 조사
- 기간·발매 유형·최근 N개 발매작 조사
- 특정 발매작 상세 조사와 발매작 간 비교

세 모드는 아티스트·발매작 식별, 데이터 수집, 수치 검증, 출처 관리가 동일하므로 별도 스킬로 나누지 않는다. 기본 진입점은 아티스트 이름이며 앨범명은 선택적인 필터 또는 상세 탐색 대상으로 사용한다.

### 3.2 기본 발매 범위

아티스트가 주 아티스트로 크레딧된 다음 발매 유형을 기본 디스코그래피에 포함한다.

- 정규 앨범
- EP·미니 앨범
- 싱글 앨범
- 디지털 싱글
- 리패키지

멤버 솔로, 피처링 참여, OST 수록곡, 라이브·컴필레이션·리믹스 앨범은 별도 범주로 보존하되 기본 성과 비교에서는 제외한다. 사용자가 요청하면 포함할 수 있다. 같은 앨범의 국가별·포맷별 판본은 발매 그룹 아래에 묶되 원본 판본과 식별자를 삭제하지 않는다.

### 3.3 장기 구조

실제 업무에서 반복되는 독립적인 질문이 확인되면 다음 전문 스킬을 추가한다.

```text
music-industry-research 플러그인
├── research-artist-discography
│   └── 아티스트의 발매작 목록, 앨범별 판매·차트·SNS, 시기별 비교
├── research-music-artist
│   └── 디스코그래피 밖의 팬덤, 채널, 시장별 현재 성과
├── analyze-music-campaign
│   └── 컴백 전후 콘텐츠와 캠페인 성과
├── scan-music-market
│   └── 장르, 국가, 플랫폼 트렌드
└── build-music-marketing-brief
    └── 조사 결과를 전략 브리프로 종합

공통 리서치 모듈
├── 아티스트·앨범 식별
├── 데이터 공급자 어댑터
├── 지표 정의와 정규화
├── 출처·신뢰도 관리
└── 표·차트용 데이터 생성
```

전문 스킬마다 별도 플러그인을 만들지 않는다. 인증, 데이터 공급자 설정, 공통 리서치 모듈을 한 플러그인에 집중해 중복과 지표 불일치를 방지한다.

## 4. 사용자 인터페이스

스킬의 외부 인터페이스는 자연어 요청 하나로 유지한다. 사용자가 공급자, 파서 또는 내부 파일 구성을 알 필요가 없어야 한다.

요청에서 다음 정보를 추론한다.

- 조사 대상 아티스트
- 전체 디스코그래피 또는 기간·유형·앨범 필터
- 관심 시장: 명시가 없으면 한국 우선, 글로벌 보조
- 관심 기간: 명시가 없으면 데뷔 발매작부터 수집 기준일까지
- 원하는 세부 지표와 출력 형식

동명의 아티스트 때문에 대상이 모호할 때만 후보와 식별 근거를 제시하고 사용자에게 선택을 요청한다. 나머지 누락 정보는 보수적인 기본값으로 진행한다. 디스코그래피가 길어도 발매작을 임의로 누락하지 않으며, 전체 목록을 먼저 보존한 뒤 본문의 상세 수준을 조절한다.

## 5. 스킬 동작 흐름

1. 요청에서 아티스트, 조사 기간, 발매 유형, 상세 탐색 대상을 추출한다.
2. 공식 식별자와 소속·국가·대표 발매작을 사용해 아티스트를 판별한다.
3. 여러 카탈로그 출처에서 발매 후보를 수집해 누락 여부를 교차 검증한다.
4. 정규·EP·싱글 앨범·디지털 싱글·리패키지 등으로 분류하고 비기본 발매를 별도 범주로 보존한다.
5. 제목, 아티스트, 발매일, 국가, UPC/EAN, MBID 등으로 동일 발매 그룹과 판본을 판정한다.
6. 각 발매작을 공식 데이터 공급자부터 병렬적으로 조사한다.
7. 판매량, 차트, SNS 관측값을 공통 스키마로 변환한다.
8. 공식 소개문·보도자료·인터뷰에서 발매작별 콘셉트와 프로모션 정보를 추출한다.
9. 출처 간 불일치, 디스코그래피 누락, 수집 시점 차이를 검증한다.
10. 근거가 있는 사실과 AI의 해석을 분리해 디스코그래피 보고서를 작성한다.
11. 비교 가능한 시계열 데이터가 있을 때만 앨범 간 변화 시각화를 만든다.

## 6. 깊은 모듈과 내부 이음매

첫 스킬은 복잡한 수집 과정을 작은 외부 인터페이스 뒤에 숨기는 깊은 모듈이어야 한다.

### 6.1 리서치 오케스트레이터

자연어 요청에서 아티스트 중심 조사 명세를 만들고 아래 내부 모듈을 조합한다. 호출자는 공급자별 호출 순서와 실패 복구를 알 필요가 없다.

### 6.2 엔터티 판별 모듈

아티스트, 발매 그룹, 국가·형태별 실제 발매판을 구분한다. 먼저 아티스트를 안정적으로 식별한 뒤 여러 출처의 발매 목록을 합쳐 디스코그래피를 구성한다. 앨범명 문자열만으로 판매량을 연결하지 않는다. UPC/EAN, MBID, Spotify ID 같은 식별자가 있으면 우선 사용하고, 없을 때 제목·아티스트·발매일 유사도를 사용한다.

### 6.3 데이터 공급자 이음매

MusicBrainz, Wikidata, Circle Chart, Oricon, YouTube처럼 실제로 여러 공급자가 존재하므로 공급자 어댑터 이음매를 둔다. 각 어댑터는 원본 응답과 정규화된 관측값을 반환한다. 테스트에서는 저장된 응답 어댑터를 사용한다.

### 6.4 근거와 관측값 모듈

모든 정형·비정형 출처를 동일한 근거 형식으로 보존한다. 보고서 작성자는 원본 사이트 구조를 알 필요 없이 관측값과 근거만 사용한다.

### 6.5 보고서 작성 모듈

디스코그래피 개요, 발매작 비교, 특정 발매작 상세 템플릿을 선택하고 사실·불확실성·해석을 구분한다. 내부 공급자가 추가되어도 보고서 인터페이스는 유지한다.

## 7. 데이터 공급자 우선순위

| 데이터 | 1순위 | 2순위 | 참고값 |
|---|---|---|---|
| 앨범 식별·발매 정보 | MusicBrainz, Wikidata | Spotify, Apple iTunes Search | 소속사·배급사 공식 페이지 |
| 한국 판매·출하 | Circle Album·Retail Chart | 공식 인증·보도자료 | 기사·위키 |
| 일본 판매 | Oricon | 공식 인증·보도자료 | 기사·위키 |
| 미국·글로벌 | Billboard 등 공식 차트 | Spotify·Apple 공개 순위 | 기사 |
| 영상 성과 | YouTube Data API | YouTube 공개 페이지 | 기사 |
| 공개 화제성 | Circle Social Chart | 접근 가능한 공식 SNS 지표 | 기사·위키 |
| 콘셉트·프로모션 | 소속사·배급사 공식 소개 | 인터뷰·보도자료 | 위키·리뷰 |

### 7.1 실제 조사에서 확인한 제약

- [Spotify 개발 모드](https://developer.spotify.com/documentation/web-api/tutorials/february-2026-migration-guide)는 2026년부터 앱 소유자 Premium, 사용자 수 제한, 일부 성과 필드 제거가 적용된다. Spotify는 앨범 식별과 트랙 목록 보조 공급자로 취급한다.
- [Apple iTunes Search API](https://performance-partners.apple.com/search-api)는 인증 없이 앨범명, 발매일, 장르, 트랙 수 같은 카탈로그 정보를 보완할 수 있다. 검색 결과의 미리보기와 콘텐츠는 Apple의 사용 조건을 따르며 판매 성과 데이터로 해석하지 않는다.
- [MusicBrainz API](https://musicbrainz.org/doc/MusicBrainz_API)는 키 없이 사용할 수 있지만 초당 1회 제한과 비상업적 무료 조건이 있다. 상업 배포 시 계약을 다시 확인한다.
- [Wikidata](https://www.wikidata.org/wiki/Wikidata%3AData_access)는 CC0 정형 데이터 공급자로 사용하지만 K-pop 데이터의 누락과 잘못된 유형을 검증한다.
- Circle Chart 공개 페이지가 호출하는 JSON 요청에서 앨범명, 아티스트, 순위, 수량, 유통사 필드를 확인했다. 이는 문서화된 공개 API가 아니므로 개인 프로토타입의 저빈도 검증 경로로만 사용하며 제품화 전 [데이터 제휴](https://circlechart.kr/page_chart/album.circle?serviceGbn=&termGbn=week)를 확인한다.
- [Oricon 공개 순위](https://www.oricon.co.jp/rank/)의 HTML에서 주간 추정 판매량을 확인했다. 상업적 재사용과 대량 수집은 법인용 데이터 조건을 확인한다.
- [YouTube Data API](https://developers.google.com/youtube/v3/docs)는 공개 영상·채널 지표의 기본 공급자다. 검색과 조회 결과를 캐시해 [할당량](https://developers.google.com/youtube/v3/determine_quota_cost)을 아낀다.
- Instagram은 공개 데이터 첫 버전의 필수 공급자로 삼지 않는다. 소유 계정이나 Professional 계정 연결은 이후 인증 플러그인 단계에서 [공식 Instagram 인터페이스](https://www.postman.com/meta/instagram/folder/23987686-3a75357f-e106-47ef-a8d9-af1aadf85365)로 추가한다.
- TikTok Display API는 사용자 승인이 필요하고 Research API는 상업적 마케팅 용도에 맞지 않는다. 첫 버전은 공식적으로 접근 가능한 공개 증거만 사용한다.
- X의 공식 인터페이스는 읽기에도 과금되므로 무료 첫 버전의 기본 공급자에서 제외한다.

## 8. 지표 의미 보존

다음 값은 모두 `판매량`으로 합치지 않는다.

- `shipment_net_returns`: 출하량에서 반품량을 뺀 값
- `retail_sale`: 소매점 실판매량
- `estimated_sale`: 표본을 기반으로 추정한 판매량
- `album_equivalent_unit`: 스트리밍과 트랙 판매를 환산한 앨범 단위
- `chart_rank`: 수량이 아니라 순위
- `certification_threshold`: 실제 누적값이 아니라 인증 기준 통과 사실

비교 보고서는 같은 시장, 같은 측정 방식, 같은 기간의 관측값만 직접 비교한다. 정의가 다른 수치는 나란히 제시할 수 있지만 합산하거나 우열 점수로 변환하지 않는다.

## 9. 정규화 데이터 형식

최소 관측값 형식은 다음과 같다.

```json
{
  "artist_id": "canonical-artist-id",
  "release_group_id": "canonical-release-group-id",
  "edition_id": "optional-country-or-format-edition-id",
  "release_type": "studio_album",
  "metric": "album_units",
  "value": 1250000,
  "unit": "copies",
  "measurement_type": "shipment_net_returns",
  "market": "KR",
  "period_start": "2026-01-01",
  "period_end": "2026-01-31",
  "observed_at": "2026-07-19T12:00:00+09:00",
  "source_tier": "A",
  "source_name": "Circle Chart",
  "source_url": "https://example.com/source",
  "evidence": "원문에서 이 값을 확인할 수 있는 짧은 근거",
  "confidence": 0.98
}
```

`source_tier`는 다음 기준을 따른다.

- A: 공식 인터페이스, 공식 차트, 소속사·배급사 공식 자료
- B: 신뢰할 만한 언론 또는 독립된 복수 출처가 일치
- C: 위키, 커뮤니티, 단일 비공식 출처
- `unverified`: 근거 부족 또는 출처 충돌

신뢰도 점수는 출처 등급을 대신하지 않는다. 점수와 등급을 함께 표시하며 자동 계산 점수를 사실처럼 표현하지 않는다.

## 10. AI가 담당하는 일과 담당하지 않는 일

AI는 다음 작업에 사용한다.

- 자연어 요청을 조사 명세로 변환
- 공식 소개문과 인터뷰에서 콘셉트·포지셔닝·프로모션 특징 추출
- 서로 다른 출처의 표현을 공통 개념으로 정리
- 수치가 뒷받침하는 범위 안에서 마케팅 시사점 작성

AI는 다음 작업을 임의로 수행하지 않는다.

- 판매량이나 SNS 수치를 추정해 빈칸 채우기
- 출하량과 실판매량 합산
- 출처에 없는 앨범 콘셉트 단정
- 서로 다른 발매판 자동 병합 후 원본 삭제
- 접근 제한 우회 또는 로그인 세션 탈취

수치 추출과 식별자 매칭은 가능한 한 결정적 스크립트로 처리하고, AI 해석에는 항상 근거를 연결한다.

## 11. 자격 증명과 무료 폴백

첫 버전은 자격 증명이 전혀 없어도 동작해야 한다.

- 자격 증명이 있으면 공식 정형 인터페이스를 우선 호출한다.
- 없으면 허용된 공식 공개 페이지와 웹 검색으로 폴백한다.
- 어느 경로에서도 찾지 못하면 `데이터 없음` 또는 `확인 불가`로 반환한다.

향후 지원할 환경 변수 이름은 다음과 같다.

```text
SPOTIFY_CLIENT_ID
SPOTIFY_CLIENT_SECRET
YOUTUBE_API_KEY
DISCOGS_TOKEN
```

자격 증명을 파일에 기록하거나 Git에 커밋하지 않는다. 로그와 오류 메시지에도 비밀값을 출력하지 않는다.

## 12. 기본 출력 형식

### 12.1 아티스트 디스코그래피 브리프

1. 핵심 요약과 주요 시사점
2. 전체 발매작 목록과 발매 유형·기준일
3. 앨범별 발매일, 장르, 트랙 수, 유통사
4. 앨범별 콘셉트와 포지셔닝 변화
5. 판매·차트 관측값과 측정 정의
6. YouTube와 공개 SNS 대체 지표
7. 프로모션 방식과 컴백 타임라인
8. 시기별 성장·변화와 마케팅 관점의 해석
9. 누락·상충 데이터와 신뢰도
10. 원문 링크

### 12.2 발매작 비교와 상세 탐색

1. 요청된 기간·유형에 해당하는 발매 조건 비교표
2. 동일 측정 방식·동일 기간 성과 비교
3. YouTube 반응 비교
4. 콘셉트와 프로모션 방식 차이
5. 시장·팬덤 규모 차이에 대한 해석 주의사항
6. 공통점, 차이점, 실행 가능한 시사점
7. 사용자가 지정한 앨범의 상세 브리프
8. 출처와 누락 데이터

### 12.3 시각화

데이터가 충분할 때만 다음 시각화를 만든다.

- 판매량 또는 차트 순위의 시간 변화
- 발매작별 발매 후 동일 기간 기준 YouTube 조회수
- 앨범별 동종 지표 비교 막대그래프
- 전체 디스코그래피 발매 타임라인
- 선택한 발매작의 티저, 뮤직비디오, 차트 진입 타임라인

서로 단위가 다른 지표를 하나의 축에 겹치거나 근거 없는 종합 점수를 만들지 않는다. 기본 결과는 대화 안에서 표시하고, 사용자가 요청하면 정규화 데이터를 CSV 또는 JSON으로 내보낸다.

## 13. 오류와 누락 처리

- 동명의 아티스트·앨범: 후보와 식별 근거를 보여주고 신뢰도가 낮으면 사용자에게 확인한다.
- 디스코그래피 출처 간 누락: 출처별 발매 목록의 차이를 기록하고 하나의 출처만으로 전체 목록을 확정하지 않는다.
- 발매 유형 불일치: 원본 분류를 보존하고 스킬의 공통 유형으로 정규화한 근거를 표시한다.
- 여러 판본: 발매 그룹과 실제 판본을 모두 보존하고 판본별 수치의 범위를 표시한다.
- 출처 충돌: 하나를 선택해 덮어쓰지 않고 각각의 정의·기준일과 함께 표시한다.
- 발매일 차이: 최초 발매일과 국가별 발매일을 분리한다.
- 크롤링 실패: 재시도 횟수를 제한하고 다른 허용 출처로 전환한다.
- 요청 제한: `Retry-After`를 지키고 저장된 캐시를 사용한다.
- 오래된 SNS 값: 수집 시각을 크게 표시하고 최신값으로 표현하지 않는다.
- 기사 표현만 있는 경우: `밀리언셀러` 같은 표현을 정확한 수량 필드에 넣지 않는다.

## 14. 나무위키와 비정형 웹 자료 정책

나무위키는 핵심 데이터베이스가 아니라 참고 출처다. 현재 `robots.txt`는 `/w/` 경로를 허용하지만 페이지 구조와 Cloudflare 동작이 바뀔 수 있고, 콘텐츠의 비영리·동일조건 라이선스가 회사 활용과 충돌할 수 있다.

- 소량의 참고 조사에만 사용한다.
- 원문 링크와 짧은 근거를 보존한다.
- 공식 출처와 다른 수치는 C 등급 참고값으로 분리한다.
- 대량 복제, 원문 저장소 구축, 접근 제한 우회는 하지 않는다.
- 회사 업무로 확장하기 전 법무와 데이터 이용 조건을 검토한다.

비정형 페이지는 정적 HTML 파서를 먼저 사용하고, JavaScript 렌더링이 필요한 경우 Playwright 또는 Crawl4AI를 제한적으로 사용한다. LLM 기반 추출 결과는 스키마 검증과 원문 근거 확인을 통과해야 한다.

## 15. 스킬 파일 구성

아래는 공급자별 파싱과 테스트가 성장했을 때의 목표 구조다.

```text
research-artist-discography/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
│   ├── collect-discography-data.py
│   ├── classify-release-types.py
│   ├── match-releases.py
│   └── providers/
│       ├── musicbrainz.py
│       ├── wikidata.py
│       ├── circle_chart.py
│       ├── oricon.py
│       └── youtube.py
└── references/
    ├── source-policy.md
    ├── data-schema.md
    └── report-format.md
```

`SKILL.md`에는 핵심 조사 절차와 어떤 참고 파일을 언제 읽어야 하는지만 둔다. 상세 공급자 제약, 데이터 스키마, 보고서 템플릿은 `references/`로 분리해 불필요한 컨텍스트 로드를 줄인다.

첫 구현에서 실제로 필요하지 않은 공급자 파일은 만들지 않는다. 공급자 로직이 짧으면 하나의 수집 스크립트 안에서 시작하고, 두 번째 구현이나 독립 테스트가 필요해질 때 어댑터 파일로 분리한다.

## 16. 검증 전략

### 16.1 결정적 테스트

- 동명의 아티스트 판별과 아티스트 식별자 보존
- 여러 카탈로그 출처의 디스코그래피 합집합과 중복 제거
- 정규·EP·싱글 앨범·디지털 싱글·리패키지 분류
- 멤버 솔로·피처링·OST·라이브·컴필레이션의 기본 비교 제외
- 동명의 아티스트와 앨범 판별
- 디지털판, 일반판, 한정판 묶기
- UPC/EAN과 MBID 우선 매칭
- Circle Chart와 Oricon 저장 응답 파싱
- 측정 방식이 다른 판매량의 비합산
- 기준일과 출처 등급 보존
- 비밀값 로그 미출력

### 16.2 보고서 행동 테스트

- 아티스트 이름만 있는 요청이 전체 디스코그래피 목록을 먼저 구성
- 디스코그래피가 길어도 발매작을 임의로 누락하지 않음
- 기간·유형 필터가 원본 전체 목록을 삭제하지 않음
- 두 개 이상 발매작을 비교할 때 동종 지표만 직접 비교
- 특정 앨범 요청이 디스코그래피 안에서 상세 브리프로 연결
- 출처가 없는 수치를 생성하지 않음
- 공식값과 위키 참고값을 구분
- 데이터가 부족하면 시각화를 생략하고 이유를 설명

### 16.3 실제 호출 스모크 테스트

- 자격 증명 없음
- MusicBrainz와 Wikidata 검색
- Circle Chart 공개 페이지의 저빈도 조회
- Oricon 공개 페이지의 저빈도 조회
- 선택적 YouTube 키 사용
- 요청 제한, 타임아웃, 사이트 구조 변경

실제 사이트 호출은 기본 테스트와 분리한다. 외부 장애가 스킬의 결정적 테스트를 실패시키지 않게 한다.

## 17. 성공 기준

첫 버전은 다음 조건을 모두 만족하면 성공으로 본다.

1. 사용자가 아티스트 이름만 입력하고 코드나 공급자 이름을 지정하지 않아도 조사를 시작할 수 있다.
2. 아티스트 이름만으로 기본 범위의 전체 디스코그래피를 식별한다.
3. 정규·EP·싱글 앨범·디지털 싱글·리패키지를 구분하고 판본 중복을 제어한다.
4. 전체 발매작 조사, 기간·유형 필터, 특정 앨범 상세 탐색을 모두 처리한다.
5. 각 핵심 수치에 출처, 측정 정의, 기간, 수집 기준일이 있다.
6. 출하량, 실판매량, 추정 판매량, 앨범 환산 단위를 혼합하지 않는다.
7. 공식 출처와 비공식 참고값을 명확히 구분한다.
8. 자격 증명 없이도 최소한 발매작 목록, 앨범 기본 정보와 공개 차트 근거를 조사한다.
9. 데이터가 부족하거나 충돌할 때 추측하지 않고 제한을 설명한다.
10. 결과가 대화 안에서 읽기 좋은 디스코그래피 브리프와 비교표로 제공된다.
11. 공급자 모듈을 교체해도 스킬의 사용자 인터페이스와 보고서 형식이 유지된다.

## 18. 첫 버전의 비목표

- 항상 실행되는 서버 또는 독립 웹 대시보드
- 매일 자동 수집하는 스케줄러
- Instagram, TikTok, X의 비공개 또는 인증 데이터
- 회사 내부 데이터웨어하우스 연결
- Luminate, Chartmetric, Soundcharts 같은 유료 계약 연동
- 모든 차트의 전체 역사 데이터 구축
- 멤버별 솔로 디스코그래피의 자동 병합
- 판매량 예측 모델
- 출처가 없는 AI 추정치

이 비목표는 실제 업무 검증 후 `music-industry-research` 플러그인의 후속 전문 스킬 또는 공통 리서치 모듈로 추가한다.
